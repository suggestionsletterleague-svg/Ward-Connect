# Ward Connect

A mobile-first **Progressive Web App** for an LDS Church ward. Members can view the
sacrament meeting program, read announcements, browse the calendar, sign up to serve,
feed the missionaries, sign up for building cleaning, view lesson schedules, and submit
private help requests — **no account required**. Only ward leaders sign in to manage content.

Built with **React 19 · Vite · React Router · Supabase (Auth + RLS) · Tailwind CSS** and
installable as a PWA on iPhone and Android.

---

## 1. Quick start

```bash
npm install
cp .env.example .env      # then fill in your Supabase values (see step 3)
npm run dev               # http://localhost:5173
```

The UI loads even before Supabase is configured, so you can preview the design
immediately. Live data appears once your credentials and database are in place.

---

## 2. Set up the Supabase database

1. Create a project at [supabase.com](https://supabase.com).
2. Open **SQL Editor → New query**, paste the contents of
   [`supabase/migrations/0001_initial_schema.sql`](supabase/migrations/0001_initial_schema.sql),
   and run it. This creates all 12 tables, triggers, and Row Level Security policies.
   Then run [`supabase/migrations/0002_capacity_enforcement.sql`](supabase/migrations/0002_capacity_enforcement.sql)
   to enforce signup limits at the database level (prevents two people claiming the
   last spot, or a meal getting a second host).
3. Create your first leader account under **Authentication → Users → Add user**
   (set an email + password). A matching `profiles` row with role `admin` is created
   automatically by a trigger.
4. *(Optional)* Promote yourself to super admin:
   ```sql
   update public.profiles set role = 'super_admin' where email = 'you@example.com';
   ```
5. **Important — turn off public sign-ups** so members can't create logins:
   **Authentication → Sign In / Providers → Email →** disable "Allow new users to sign up".

### Tables created
`profiles`, `programs`, `announcements`, `calendar_events`, `volunteer_opportunities`,
`volunteer_signups`, `missionary_meals`, `missionary_meal_signups`,
`building_cleaning_slots`, `building_cleaning_signups`, `lesson_schedules`, `help_requests`.

### Security model (RLS)
- **Anyone (anon)** can *read* published programs, announcements, events, lessons,
  volunteer opportunities, missionary meal slots, and cleaning slots.
- **Anyone** can *insert* volunteer / meal / cleaning signups and help requests.
- **Signup details and help requests are never publicly readable** — only admins can
  read them. The public sees remaining-spot counts via trigger-maintained counter
  columns, never the names/phone/email of who signed up.
- **Only admins** (role `admin` or `super_admin`) can create, edit, or delete official
  content and view private submissions.

## 3. Environment variables

Create a `.env` file (copy from `.env.example`):

```
VITE_SUPABASE_URL=https://your-project-ref.supabase.co
VITE_SUPABASE_ANON_KEY=your-anon-public-key
```

Find both under **Supabase → Project Settings → API**. The anon key is safe to expose
in a client app *because* RLS enforces all access rules above.

---

## 4. Admin area

Visit **`/admin`** (or tap "Ward leader sign in" at the bottom of the Home screen) and
sign in with the account you created. Admins can manage every section: draft & publish
sacrament programs (with preview), post announcements and events, open volunteer / meal /
cleaning slots, set lesson schedules, view private help requests, and remove invalid
signups. A leader who forgets their password can use **"Forgot your password?"** on the
sign-in screen to receive an emailed reset link (requires the redirect URL above).

## 5. Deploy to Netlify

This repo includes [`netlify.toml`](netlify.toml) with the correct build settings and an
SPA redirect, so deployment works without changes.

1. Push the project to GitHub/GitLab.
2. In Netlify: **Add new site → Import an existing project** and pick the repo.
   (Build command `npm run build` and publish directory `dist` are auto-detected from
   `netlify.toml`.)
3. Under **Site configuration → Environment variables**, add:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
4. Deploy. Done.

> Tip: in **Supabase → Authentication → URL Configuration**, set **Site URL** to your
> Netlify URL and add `https://your-site.netlify.app/admin/reset` to **Redirect URLs**
> so admin login and the password-reset link both work in production.

## 6. PWA / installation

- Manifest, service worker, and offline fallback are generated by `vite-plugin-pwa`.
- On Android (Chrome) an **Install** banner appears; on iOS use Safari → Share →
  **Add to Home Screen**.
- The service worker is disabled in `npm run dev` and active in production builds.
  To test it locally: `npm run build && npm run preview`.

App name **Ward Connect** · short name **Ward** · theme `#1e3a5f` · background `#ffffff`.

---

## 7. Customizing for your ward

- Ward name & tagline: `src/lib/constants.js` (`WARD_NAME`, `WARD_TAGLINE`).
- Categories and colors: also in `src/lib/constants.js`.
- App icons: replace `public/icon-192.png` and `public/icon-512.png`.
- Colors & fonts: `tailwind.config.js`.
